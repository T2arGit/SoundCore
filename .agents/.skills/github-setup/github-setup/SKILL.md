# Навык настройки GitHub

## Рабочий процесс: Первая настройка репозитория

### Этап 1: Сбор информации (спросите пользователя)

Перед началом спросите следующие вопросы (сгруппируйте их в одном сообщении):

1. **Имя репозитория**: Как должен называться репозиторий на GitHub?
2. **Видимость**: Публичный или приватный?
3. Вам понадобится их **Персональный токен доступа (PAT)** GitHub, если он еще не задан в окружении. Скажите им: перейдите на [github.com/settings/tokens](https://github.com/settings/tokens) → Generate new token (classic) → отметьте область `repo` → скопируйте его.

НЕ спрашивайте имя пользователя GitHub — его можно узнать из `git config --global user.name` или они сами могут его назвать.

### Этап 2: Запуск скрипта `check_gitignore.py`

Всегда запускайте этот скрипт первым. Он автоматически определяет тип проекта и добавляет безопасные исключения, но все равно требует проверки перед фиксацией (commit):

```bash
python .agents/.skills/github-setup/github-setup/scripts/check_gitignore.py --path <project_root>
```

Скрипт выполнит:

- Определение стека (Node.js, Rust/Tauri, Python и т.д.)
- Автоматическое добавление безопасных исключений для локальной среды/времени выполнения, таких как `.neira_runtime/`, временные файлы и папки с тяжелыми зависимостями.
- Избежание слепого скрытия навыков или документации, принадлежащих репозиторию.
- Вывод сводки того, что было добавлено/обновлено.

Перед принятием игнорирования проверьте особенно эти директории:

- `.agents/`
- `.agents/.skills/`
- `neira/skills/`
- `.deepagents/`

В некоторых проектах намеренно фиксируются навыки и конфигурации агентов.
Обсудите с пользователем и внесите корректировки перед фиксацией.

### Этап 3: Проверка README.md

- Если `README.md` не существует, создайте профессиональный файл, используя описание проекта.
- **ОБЯЗАТЕЛЬНО**: Всегда создавайте README на двух языках в одном файле: сначала английская версия, затем русская.
- Включите: название проекта, описание, ключевые функции, стек технологий, инструкции по настройке.

### Этап 4: Инициализация и Пуш

Запустите эти команды (замените переменные):

```bash
# 1. Инициализация git репозитория (если еще не инициализирован)
git init
git branch -M main

# 2. Индексация всех файлов
git add .
git status  # Проверьте, что именно фиксируется

# 3. Первый коммит
git commit -m "Начальный коммит: <название проекта>"

# 4. Создание GitHub репозитория через API (требуется PAT)
# В Windows используйте PowerShell (см. scripts/create_github_repo.ps1)
```

### Этап 5: Настройка учетных данных (если нужно)

Если при пуше запрашивается пароль, настройте хранилище учетных данных:

```bash
git config --global credential.helper store
```

Затем выполните пуш один раз — он запросит имя пользователя + PAT в качестве пароля и запомнит их.
Либо сохраните PAT в Диспетчере учетных данных Windows:

```bash
git config --global credential.helper manager
```

## Рабочий процесс: Ежедневный безопасный пуш

Используйте скрипт `safe_push.ps1` — он пошагово проведет вас через процесс и предупредит, если что-то пойдет не так.

```powershell
.\.agents\.skills\github-setup\github-setup\scripts\safe_push.ps1
```

Он выполнит:

1. Отображение текущей ветки (предупредит, если вы на `main`)
2. Показ измененных файлов
3. Запрос описания изменений (commit message)
4. Автоматический коммит и пуш
5. Напоминание о том, как выполнить слияние в `main`, когда будете готовы

### Стратегия веток (для одного разработчика)

```
main  → всегда стабильная ветка, то, что люди видят на GitHub
dev   → ветка для повседневной работы
```

**Настройка ветки dev (один раз на проект):**

```powershell
git checkout -b dev
git push -u origin dev
```

**Ежедневный рабочий цикл:**

1. Работаете над кодом как обычно.
2. Запускаете `safe_push.ps1` → он подхватывает изменения, делает коммит и пушит в `dev`.
3. Когда функция готова и протестирована → слияние в `main`:

```powershell
git checkout main
git merge dev
git push origin main
git checkout dev   # возврат в dev
```

**Правило:** Никогда не делайте `--force` push в `main` после первой настройки.

## Примечания

- **PAT вместо пароля**: Когда Git запрашивает пароль, используйте Персональный токен доступа (PAT), а НЕ пароль от аккаунта GitHub.
- **Область токена**: Минимально необходимая область — `repo` для приватных репозиториев или `public_repo` для публичных.
- **Хранение токена**: Никогда не фиксируйте токен в репозитории. Храните его только в переменных окружения или в Диспетчере учетных данных Windows.
- **`.gitignore` всегда первым**: Никогда не делайте коммит перед запуском `check_gitignore.py`.
